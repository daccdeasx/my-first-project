# Movie_demo 接口静态测试清单

> 说明：本清单基于后端 `urls.py/router/@api_view` 静态梳理接口路径与鉴权方式。建议你用 Postman 或 `curl` 进行“请求构造 + 状态码/字段校验”的静态测试。
>
> Base URL：`http://localhost:8000/api`

## 0. 通用约定

1. 鉴权头（TokenAuthentication）
   - `Authorization: Token <token>`
2. 常见断言（静态测试都可复用）
   - 未登录/无 Token：应返回 `401` 或 `403`（视视图权限策略）
   - 入参缺失/类型不对：应返回 `400`
   - 资源不存在：应返回 `404`
   - 服务器侧异常：应返回 `5xx`
3. 分页（如使用 DRF PageNumberPagination）
   - 查询参数通常包含：`page`、`page_size`（若未声明则使用默认）
   - 响应通常包含：`count/next/previous/results`（或项目中自定义的 `page/total_pages/results`）

## 1. 电影列表浏览（前台首页/列表）

### 1.1 TMDB 代理列表接口（now_playing/top_rated/popular/featured/discover）

- `GET /movies/tmdb/?type={now_playing|top_rated|popular|featured|discover}&page=1`
- 鉴权：无需 Token（后端未显式要求）

静态测试用例：
1. 成功：`type=now_playing&page=1` 期望 `200`，且返回 JSON 中包含 `results`、`page`、`total_pages`
2. 参数校验：`page=0` 期望 `400`
3. 错误类型：`type=unknown` 期望 `400`
4. 兜底：当 TMDB 不可用时，后端可能回退到本地数据，期望仍返回 `200`（或返回 `502/503` 但不应抛 HTML）

示例：
```bash
curl -i "http://localhost:8000/api/movies/tmdb/?type=popular&page=1"
```

### 1.2 电影搜索（tmdb_proxy type=search）

- `GET /movies/tmdb/?type=search&query=xxx&page=1`
- 鉴权：无需 Token

静态测试用例：
1. 成功：`query=matrix&page=1` 期望 `200` 且 `results` 为数组
2. 空查询：`query=` 期望 `200` 或 `400`（以 TMDB/后端处理为准）

示例：
```bash
curl -i "http://localhost:8000/api/movies/tmdb/?type=search&query=matrix&page=1"
```

## 2. 电影详情

- `GET /movies/{movie_id}/`
- 鉴权：无需 Token（但互动字段会根据是否登录返回不同内容）
- `movie_id`：后端实现中将其当作 `tmdb_id`

静态测试用例：
1. 成功：`movie_id` 为存在的 TMDB id，期望 `200` 且返回字段：
   - `detail`、`credits`、`similar`、`interaction`
2. 不存在：给一个无效 id，期望 `404`（或在同步逻辑失败时出现 `5xx`）
3. TMDB 不可用：期望返回 `5xx` 且 JSON 结构含 `error/code`

示例：
```bash
curl -i "http://localhost:8000/api/movies/603/"
```

## 3. 推荐模块

### 3.1 基础推荐（无需登录校验）

- `GET /recommend/`
- 鉴权：后端未显式加权限类（建议测试未登录也应可访问）

期望：
- `200`，响应包含：
  - `status`、`count`、`recommendations`（数组，每项包含 `tmdb_id/title/poster_path/vote_average`）

示例：
```bash
curl -i "http://localhost:8000/api/recommend/"
```

### 3.2 LLM 推荐（需 Token）

- `POST /llm/recommend/`
- 鉴权：`401/403`（无 Token/无权限时）
- Body（示例）：
```json
{ "movie_ids": [603, 101] }
```

期望：
- 成功 `200/201`（以实现为准），响应包含 `reasons`
- 参数缺失：可能返回 `500/400`（建议你用空 body 测一次确认）

示例：
```bash
curl -i -X POST "http://localhost:8000/api/llm/recommend/" \
  -H "Authorization: Token $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"movie_ids\":[603,101]}"
```

### 3.3 AI 聊天（需 Token）

- `POST /llm/chat/`
- Body（示例）：
```json
{ "message": "给我推荐类似的电影", "history": [] }
```

期望：
- `400`：`message` 为空或超长
- 成功：返回 `movies`、`updated_history`

示例：
```bash
curl -i -X POST "http://localhost:8000/api/llm/chat/" \
  -H "Authorization: Token $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"message\":\"给我推荐类似的电影\",\"history\":[]}"
```

## 6. 购票与订单（用户侧）

> 订单相关是 DRF ModelViewSet + router 默认路由
> 因此 Base 为 `/api/` 下的：
> - `/cinemas/`
> - `/schedules/`
> - `/orders/`

### 6.1 影院

- `GET /cinemas/` 列表
- `GET /cinemas/{id}/` 详情
- `GET /cinemas/nearby/?lat=...&lng=...` 附近影院（实现为简单返回全量）

静态测试用例：
1. 列表 `200`
2. nearby `200` 且返回数组

示例：
```bash
curl -i "http://localhost:8000/api/cinemas/nearby/?lat=31.2&lng=121.4"
```

### 6.2 场次（可过滤）

- `GET /schedules/`
- 查询参数（实现支持）：
  - `movie={movie_id}`
  - `cinema={cinema_id}`
  - `date=YYYY-MM-DD`（按 start_time__date 过滤）

静态测试用例：
1. 不带参数：`200` 返回列表
2. 带 `movie/cinema/date`：`200` 且列表过滤生效（字段层面可抽检）
3. date 格式错误：预期 `400` 或 `500`（建议确认实现对异常的处理）

示例：
```bash
curl -i "http://localhost:8000/api/schedules/?movie=603&cinema=1&date=2026-03-04"
```

### 6.3 下单 / 订单详情

- `GET /orders/`（只返回当前用户；staff 返回全部）
- `POST /orders/` 创建订单（购票）
- `GET /orders/{id}/` 详情
- `PUT/PATCH /orders/{id}/`（如允许）
- `DELETE /orders/{id}/`（如允许）
- 额外动作（仅 POST）：
  - `POST /orders/{id}/cancel/`
  - `POST /orders/{id}/change/`

鉴权：需要 Token

`POST /orders/` Body 重点字段（按实现）：
- `schedule`：场次 id；也可能传 mock/fallback 前缀触发“模拟订单”（order.schedule=None，status=paid）
- `seats`：座位数组
- `movie_id`、`cinema_id`：用于模拟订单增强 seat 信息
- `total_price`：可选（真实场次会按 `schedule.price * len(seats)` 计算）

静态测试用例：
1. 创建真实订单：传存在的 `schedule` 且 `seats` 不重复占用 → `201`
2. 座位已占用：期望 `400` 且 error 文本包含“已被占用”
3. 模拟订单：传 `schedule` 为 `null` 或以 `mock_`/`fallback_` 开头 → `201` 且 `status=paid`
4. cancel：仅当 `order.status=paid` 才成功，否则 `400`
5. change：仅当 `order.status=paid` 才成功，否则 `400`

示例（创建订单-真实场次）：
```bash
curl -i -X POST "http://localhost:8000/api/orders/" \
  -H "Authorization: Token $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"schedule\":1,\"seats\":[\"A1\",\"A2\"]}"
```

## 8. 电影数据模块（后台）

> 电影管理：`/api/admin/movies/...`
> 备注：项目中同一路径存在两套挂载方式（一个 `movies.urls`，一个 `core.admin_router`），但实际生效通常以 `movies.urls` 为准（以你运行时路由为准）。

### 8.1 电影管理（增删改查）

- `GET /admin/movies/?page=1&page_size=20&search=xxx`
- `POST /admin/movies/`
- `PUT /admin/movies/{tmdb_id}/`
- `DELETE /admin/movies/{tmdb_id}/`
- 鉴权：需要 `IsAdminUser`（无权限期望 `401/403`）

静态测试用例：
1. 未登录/普通用户：期望 `403`
2. POST 缺字段：期望 `400`
3. POST tmdb_id 重复：期望 `400`
4. GET 列表：`200` 且返回电影数组

POST Body（按 `MovieAdminSerializer`）：
- `tmdb_id`（int，必须）
- `title`（string，必须）
- `overview`（可空）
- `poster_path`（可选）
- `release_date`（YYYY-MM-DD，必须时看前端/serializer 校验）
- `runtime/budget/revenue`（可选）
- `genres/production_countries/spoken_languages`（JSON 数组；后端为 JSONField，具体格式按你数据）

示例：
```bash
curl -i -X POST "http://localhost:8000/api/admin/movies/" \
  -H "Authorization: Token $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"tmdb_id\":603,\"title\":\"The Matrix\",\"release_date\":\"1999-03-31\",\"runtime\":136}"
```

### 8.2 导出电影数据（CSV）

- `GET /admin/movies/export/`
- 鉴权：IsAdminUser
- 期望：
  - `200`
  - `Content-Type: text/csv` 或返回 CSV 流

示例：
```bash
curl -i -L "http://localhost:8000/api/admin/movies/export/" \
  -H "Authorization: Token $TOKEN" --output /tmp/movies.csv
```

### 8.3 评论管理（后台）

- `GET /admin/reviews/?page=1&search=xxx&status=true|false`
- `DELETE /admin/reviews/{review_id}/` → 期望 `204`
- `POST /admin/reviews/{review_id}/toggle-status/`
- `POST /admin/reviews/batch/` Body：`{ "review_ids": [1,2,3] }`
- 鉴权：IsAdminUser

静态测试用例：
1. 普通用户访问：`403`
2. GET 列表：`200`
3. toggle-status：`200` 返回 `is_approved`
4. batch 删除缺 body：`400`
